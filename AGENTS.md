# CLanguage

C/C++ compiler, stack-based VM, and native code editor in C#. Targets interactive mobile apps (iOS, macOS, Mac Catalyst). Performance is a primary constraint â€” the VM runs in 100 kHz loops on iPhones. .NET 9+.

## Build

```bash
dotnet build CLanguage.sln
dotnet test CLanguageTests/CLanguageTests.csproj
```

## Regenerate Parser

The Jay parser generator (C# Yacc) produces `CLanguage/Parser/CParser.cs` from the grammar. Run from repo root:

```bash
cd CLanguage && bash prebuild.sh
```

This compiles `Jay/src/*.c` into the `jay` binary, then runs it on `CParser.jay` with `Jay/skeleton.cs` to produce the generated parser. It outputs the file `CLanguage/Parser/jay_warnings.txt` with any warnings from the grammar (e.g., shift/reduce conflicts).

```

  â†’ Lexer (Token[])
```

## Parser

### Lexer (`Parser/Lexer.cs`)

Character-by-character tokenizer. Produces `Token[]` with location tracking (document, line, column, index). Handles C comments (`//`, `/* */`), string/char literals, numeric constants, ~40 keywords. Has an `IsTypedef` hook so the parser can feed back typedef names â€” required because C's grammar is ambiguous without knowing which identifiers are type names.

### Preprocessor (`Parser/Preprocessor.cs`)

Operates on token arrays, not text. Iterative expansion â€” loops until no changes detected (macro expansions can trigger other macros).

- **`#define`**: Object-like and function-like macros. Stored as `Dictionary<string, Define>` where `Define` holds name, parameters, body tokens.
- **`#include`**: Custom callback `Token[]? Include(string path, bool relative)`. Built-in `stdint.h` returns empty. System headers defined in `MachineInfo.SystemHeadersCode`.
- **`#if`/`#ifdef`/`#ifndef`/`#else`/`#endif`**: Full nesting via depth counter. `#if` expressions are evaluated by compiling them as C code through a `PreprocessorContext` (extends `EmitContext`).
- **Gotcha**: To prevent infinite recursion, a macro's own name is removed from scope during its body expansion.

### Grammar (`Parser/CParser.jay`)

ANSI C grammar (from quut.com) extended with C++ class support (class/struct/union with visibility specifiers, member functions). ~40 token types, standard expression precedence hierarchy. Entry rule: `translation_unit`.

### Generated Parser (`Parser/CParser.cs`)

LALR parser generated by Jay. **Do not edit** â€” regenerate with `prebuild.sh`. Partial class combined with `CParserImpl.cs`.

### Parser Support (`Parser/CParserImpl.cs`)

Hand-written helpers merged with the generated parser:
- `ParseTranslationUnit()` â€” main entry, runs lexer â†’ preprocessor â†’ LALR parse.
- `AddDeclaration()` â€” processes declarations, updates typedef tracking.
- `FixPointerAndArrayPrecedence()` â€” corrects C's notoriously tricky declarator binding (e.g., `int *p[10]` vs `int (*p)[10]`).
- `ParserInput` adapts token stream to Jay's `yyInput` interface. Tracks typedef set and converts `IDENTIFIER` â†’ `TYPE_NAME` tokens on the fly.

## AST (`Syntax/`)

### Expressions (`Expression.cs` base)

All expressions implement:
- `DoEmit(EmitContext)` â€” generate bytecode pushing result onto value stack.
- `GetEvaluatedCType(EmitContext)` â€” return the C type of the expression.
- `EvalConstant(EmitContext)` â€” compile-time evaluation for const expressions.
- `CanEmitPointer` / `DoEmitPointer()` â€” for `&expr` (address-of).

Key expression types: `ConstantExpression`, `VariableExpression`, `BinaryExpression`, `FuncallExpression`, `CastExpression`, `ArrayElementExpression`, `MemberFromReferenceExpression` (`.`), `MemberFromPointerExpression` (`->`), `UnaryExpression`, `AddressOfExpression`, `DereferenceExpression`, `SizeOfExpression`.

### Statements (`Statement.cs` base)

`Block`, `IfStatement`, `WhileStatement`, `ForStatement`, `SwitchStatement`, `ReturnStatement`, `BreakStatement`, `ContinueStatement`, `ExpressionStatement`. Each emits bytecode via `DoEmit(EmitContext)`. `AlwaysReturns` property for control flow analysis.

### Declarations

C's complex declarator syntax is modeled as a recursive hierarchy in `Declarator.cs`:
- `IdentifierDeclarator` â€” base name
- `PointerDeclarator` â€” wraps inner with `*`
- `ArrayDeclarator` â€” wraps with `[N]`
- `FunctionDeclarator` â€” wraps with `(params)`

Example: `int *arr[10]` â†’ `ArrayDeclarator(PointerDeclarator(IdentifierDeclarator("arr")))`.

### Block (`Block.cs`)

Compound statement. Holds `Variables`, `Functions`, `Typedefs`, `Structures`, `Enums`, `InitStatements`. Nested blocks create scopes. `TranslationUnit` extends `Block` as the root AST node.

## Type System (`Types/`)

`CType` is the abstract base. Key property: **`NumValues`** â€” the number of `Value` slots this type occupies on the stack. This is NOT byte size; it's the count of 8-byte `Value` structs needed. A `char` takes 1 Value. An `int[10]` takes 10 Values. A struct with 3 int fields takes 3 Values.

### Type Hierarchy

| Type | `NumValues` | Notes |
|------|-------------|-------|
| `CBasicType` (base for int/float types) | 1 | Static instances: `SignedInt`, `UnsignedInt`, `Float`, `Double`, etc. |
| `CIntType` : `CBasicType` | 1 | Name + Signedness + Size. Integer promotion and arithmetic conversion methods. |
| `CFloatType` : `CBasicType` | 1 | Float (32-bit), Double (64-bit). |
| `CBoolType` : `CBasicType` | 1 | Single static instance. |
| `CPointerType` | 1 | `InnerType` property. Byte size from `MachineInfo.PointerSize`. |
| `CArrayType` | `Length Ă— ElementType.NumValues` | Array-to-pointer decay via `ScoreCastTo()`. Incomplete arrays (`[]`) have NumValues=1. |
| `CStructType` | Sum of member `NumValues` | Members laid out sequentially. `GetFieldValueOffset()` computes member offset in Value slots. |
| `CEnumType` | 1 | Acts as integer. Members have Name + int Value. |
| `CFunctionType` | 1 | `ReturnType` + `Parameters` list. `IsInstance` flag for member functions. Parameter offsets are negative (stack-relative from frame pointer). |

### Critical Design Decision: Value-Based Layout, Not Byte Layout

The stack is an array of `Value` structs. Each `Value` is 8 bytes regardless of the C type it holds. A `char` occupies one full `Value` slot, same as a `double`. Structs and arrays occupy N consecutive `Value` slots where N = number of scalar fields (recursively). **There is no byte-level memory layout.** Pointer arithmetic and `sizeof` use `MachineInfo` for byte-size calculations, but actual memory addressing is in Value-slot units. This is the most important architectural decision to understand.

### MachineInfo (`MachineInfo.cs`)

Target machine configuration: `CharSize`, `IntSize`, `PointerSize`, etc. Presets: `Windows32` (4-byte pointers), `Mac64` (8-byte pointers). Contains `InternalFunctions` (C# implementations of builtins like `printf`), `SystemHeadersCode` (math.h, etc.), and `HeaderCode` (generated typedefs like `int8_t`).

**CLR interop**: `AddGlobalReference(name, target)` and `AddGlobalMethods(target)` use reflection + `System.Linq.Expressions` to auto-marshal C# objects as C structs with callable methods. Generates C header code and `InternalFunction` wrappers at runtime.

## Compiler (`Compiler/`)

### CCompiler (`CCompiler.cs`)

Orchestrates full compilation. Constructor processes documents: lex â†’ preprocess â†’ parse each file into a `TranslationUnit`. `Compile()` does:

1. **Collect declarations** â€” scan all TranslationUnits for globals, functions, types, typedefs.
2. **Generate `__cinit()`** â€” synthetic function that runs all global initializers (per-file `__{filename}__cinit()` functions).
3. **Link** â€” populate function references so recursion/forward calls resolve.
4. **Emit** â€” compile each function body into bytecode via `FunctionContext`.

Output: `Executable` containing `Functions`, `Globals`, and `MachineInfo`.

### Emit Contexts (scoped symbol resolution)

`EmitContext` is the base. Subclasses form a chain for nested scope resolution:

```
ExecutableContext (globals, all functions)
  â””â”€â”€ TranslationUnitContext (file-local typedefs, structs, enums)
        â””â”€â”€ FunctionContext (params, locals â€” extends BlockContext)
              â””â”€â”€ BlockContext (block-scoped variables)
                    â””â”€â”€ LoopContext (break/continue labels)
```

Variable resolution walks up this chain: locals â†’ params â†’ block vars â†’ file scope â†’ globals â†’ `MachineInfo` builtins.

Key `EmitContext` methods:
- `Emit(OpCode, Value)` â€” append instruction (delegated to `FunctionContext`'s `CompiledFunction.Instructions`).
- `EmitCast(from, to)` â€” emit type conversion instruction.
- `MakeCType(specs, declarator)` â€” build `CType` from AST declaration specifiers + declarator.
- `GetInstructionOffset(CType)` â€” map type to opcode offset (Int8=0, UInt8=1, ... Float64=9) for type-parameterized instructions.
- `DefineLabel()` / `EmitLabel()` â€” control flow targets.

### Code Emission Pattern

Each AST node's `DoEmit()` pushes its result onto the value stack:
```
BinaryExpression(a + b):
  Left.Emit()        â†’ push a
  EmitCast(leftType, commonType)
  Right.Emit()       â†’ push b
  EmitCast(rightType, commonType)
  Emit(AddInt32)     â†’ pop a,b; push a+b    (opcode selected by type offset)
```

Functions are called by pushing args, then a function address constant, then `Call`. The Call opcode pops the address and enters the function.

## Virtual Machine (`Interpreter/`)

### Performance Design

The VM is designed for **zero-allocation, zero-search** hot loops. It will run at 100 kHz+ on iPhones â€” `Step(microseconds)` is called repeatedly from app render loops. Every cycle counts.

- **Pre-allocated stack**: `Value[] Stack` â€” fixed-size array, no resizing ever.
- **Pre-allocated frames**: `ExecutionFrame[] Frames` â€” fixed pool, no allocation on function call.
- **No heap, no GC**: All memory is stack-resident. Pointers are raw integer offsets into the `Value[]` array.
- **Budget-based execution**: `Step(microseconds)` decrements `RemainingTime` by `CpuSpeed` per instruction. Returns when budget exhausted. `CpuSpeed` default = 1000 (instructions per microsecond).
- **Yield support**: `Yield(value)` + `SleepTime` allow cooperative scheduling.

### Value (`Value.cs`)

**The fundamental unit of the VM.** 8-byte struct with explicit field layout â€” a union:

```csharp
[StructLayout(LayoutKind.Explicit, Size = 8)]
public struct Value {
    [FieldOffset(0)] public double Float64Value;
    [FieldOffset(0)] public long Int64Value;
    [FieldOffset(0)] public int Int32Value;
    [FieldOffset(0)] public float Float32Value;
    [FieldOffset(0)] public int PointerValue;  // same offset as Int32Value
    // ... all 10 numeric types at offset 0
}
```

Same 8 bytes, different interpretations. Implicit conversions from all C# numeric types. **Pointers are `int` offsets into the `Value[] Stack`** â€” not byte addresses. `Value.Pointer(offset)` creates a pointer value.

### CInterpreter (`CInterpreter.cs`)

```csharp
public readonly Value[] Stack;     // All memory (globals + locals + temporaries)
public int SP;                      // Stack pointer â€” next free slot
readonly ExecutionFrame[] Frames;   // Call stack (pre-allocated pool)
int FI;                             // Frame index (-1 = idle)
public int CpuSpeed = 1000;        // Instructions per microsecond budget
```

`Reset(entrypoint)` initializes: SP=0, copy global initial values to stack bottom, push first call frame. `Step(microseconds)` runs instructions within time budget. `Run()` = `Step(1_000_000)`.

### Stack Layout

```
Address 0..N:  Global variables (laid out by Executable.AddGlobal)
               String constants (char arrays in globals)
After globals: Stack grows upward with function calls

Per function call:
  [...args pushed by caller...]
  FP â†’ (frame pointer set here by Call)
  [local variables allocated by Init()]
  [expression temporaries]
  SP â†’ next free slot
```

Globals live at the bottom of the stack. Each `CompiledVariable` has a `StackOffset` in Value-slot units. Parameter offsets are negative relative to FP. Local offsets are positive relative to FP.

### Instruction Set (`Instruction.cs`)

```csharp
public struct Instruction {
    public OpCode Op;     // Operation
    public Value X;       // Immediate operand (constants, addresses)
    public Label? Label;  // Branch target
}
```

**Opcode categories:**
- **Control**: `Jump`, `BranchIfTrue/False`, `BranchIfTrue/FalseNoSPChange`, `Call`, `Return`
- **Memory**: `LoadConstant`, `LoadFramePointer`, `LoadGlobal/Arg/Local`, `StoreGlobal/Arg/Local`, `LoadPointer`, `StorePointer`, `OffsetPointer`
- **Arithmetic**: `Add/Subtract/Multiply/Divide/Modulo/ShiftLeft/ShiftRight` Ă— 10 type variants
- **Relational**: `EqualTo/LessThan/GreaterThan` Ă— 10 type variants
- **Bitwise**: `BinaryAnd/BinaryOr/BinaryXor` Ă— 10 type variants
- **Unary**: `Not/BinaryNot/Negate` Ă— 10 type variants
- **Conversion**: `ConvertXtoY` for all 100 type pairs (10Ă—10)
- **Boolean**: `LogicalAnd`, `LogicalOr`
- **Stack**: `Dup`, `Pop`

**Type parameterization**: Opcodes are grouped in sets of 10 (Int8, UInt8, Int16, UInt16, Int32, UInt32, Int64, UInt64, Float32, Float64). The compiler computes an offset 0â€“9 via `GetInstructionOffset()` and adds it to the base opcode: `OpCode.AddInt8 + offset`.

**`BranchIfFalseNoSPChange`**: Used for short-circuit evaluation (`&&`, `||`) â€” tests the top of stack without popping it.

**`OffsetPointer`**: Pops pointer + integer offset, pushes pointer + offset. Used for array indexing and struct member access (offset in Value-slot units, not bytes).

### Execution Loop (`CompiledFunction.Step()`)

Tight `while` loop over `Instructions` list. Switch on `OpCode`. Directly manipulates `state.Stack[state.SP]`. One function's Step() runs until `Call` (enters new frame, returns), `Return` (pops frame, returns), or time budget exhausted. The interpreter then re-enters the top frame's Step() on the next cycle.

### Function Types

- **`CompiledFunction`**: User C code compiled to bytecode. `Init()` allocates local variable space on stack (advances SP). `Step()` runs the instruction loop. `LocalVariables` list with stack offsets.
- **`InternalFunction`**: C# builtin. `Action` delegate receives `CInterpreter`, reads args via `state.ReadArg(i)`, pushes results via `state.Push()`. Then calls `state.Return()`. Prototypes are parsed from C strings.

### Executable (`Executable.cs`)

Container for compiled program: `Functions` (compiled + internal), `Globals` (with stack offsets and initial values). `GetConstantMemory(string)` allocates a string constant as a global char array and returns a pointer to it.

## Editor (`Editor/`)

Native UI component using `#if __IOS__` / `#elif __MACOS__` conditional compilation for UIKit vs AppKit.

- **`CEditor`** â€” Main view. Wraps `EditorTextView` (text editing), `ErrorView` (inline error display), `MarginView` (line numbers). Calls `CLanguageService.Colorize()` on text changes for syntax highlighting.
- **`Theme`** â€” Light/dark mode colors, fonts. Xcode-inspired palette. Defines `NSStringAttributes` for each syntax element (keyword, type, function, string, number, comment).
- **`EditorKeyboardAccessory`** (iOS) â€” Extra keyboard buttons for brackets, semicolons etc.
- **`CEditorController`** â€” View controller managing editor + compilation + execution lifecycle.
- **`EditorPrinter`** â€” `Report.Printer` implementation that collects errors for display in the editor.

Colorization is token-based (not semantic): lex the code, classify each token kind to a color. Function names are detected by cross-referencing with `MachineInfo.InternalFunctions` and compiled globals.

## Testing (`CLanguageTests/`)

Test infrastructure in `TestsBase.cs`:
```csharp
// Wraps user code in start() that calls __cinit() then main()
protected CInterpreter Run(string code, params int[] expectedErrors);
```

Tests write C code as strings, compile and execute, then check results or expected error codes. `TestMachineInfo` extends `MachineInfo` with `assertAreEqual(expected, actual)` for in-C assertions. `ArduinoTestMachineInfo` adds Arduino-specific builtins (digitalWrite, analogRead, etc.).

## Known Limitations

- No `goto`
- No variadic functions
- No full heap/malloc (all memory is stack-based)
- `volatile`/`restrict` parsed but ignored
- No optimization passes â€” direct bytecode emission
- Limited forward declaration support
